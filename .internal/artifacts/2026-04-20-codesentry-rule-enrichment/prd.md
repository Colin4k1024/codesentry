# PRD - CodeSentry 规则丰富与本地测试体系建设

## 背景

CodeSentry 是一个 Go 语言编写的静态代码分析工具，支持 11 种编程语言的安全规则扫描。当前项目核心功能已完成，但缺乏系统化的测试体系和充分的规则覆盖。

**触发原因：** 提升规则的可靠性和项目可维护性

**当前约束：**
- 无任何 `*_test.go` 文件
- `go test ./...` 无实际测试用例
- 规则仅基于 regex 模式，缺少 AST 深度检测的测试覆盖

---

## 目标与成功标准

### 业务目标
1. 建立完整的 Go 单元测试体系，覆盖率 ≥ 80%
2. 丰富安全规则，覆盖常见漏洞类型
3. 建立规则有效性验证机制

### 用户价值
- 开发者：获得更可靠的扫描结果
- 维护者：规则变更可被自动化验证

### 成功指标
| 指标 | 目标 |
|------|------|
| 测试覆盖率 | ≥ 80%（engine, parser, rules 核心模块）|
| 规则数量 | 每个语言至少 5 条安全规则 |
| 规则准确性 | 通过 golden file 测试验证 |

---

## 用户故事

### US-1: 测试体系建设
> 作为一个维护者，我希望运行 `go test ./...` 能验证规则正确性，这样规则变更不会被引入 bug。

**验收标准：**
- [ ] `go test ./...` 通过率 100%
- [ ] 覆盖率报告生成成功
- [ ] 关键路径（Scan, Parse, Rule matching）被覆盖

### US-2: 安全规则丰富
> 作为一个安全工程师，我希望能检测更多类型的漏洞，这样我可以更全面地保护代码库。

**验收标准：**
- [ ] 每种语言至少 5 条安全规则
- [ ] 常见漏洞类型（注入、硬编码密钥、敏感数据泄露）全覆盖
- [ ] 规则有对应的测试用例

### US-3: 规则质量验证
> 作为一个 QA，我希望能验证规则的准确性，这样我可以确保扫描结果可信。

**验收标准：**
- [ ] Golden file 测试验证 regex 模式正确性
- [ ] 规则误报率 < 5%
- [ ] 规则漏报率文档化

## 风险与依赖

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 测试数据管理复杂 | 维护成本高 | 使用 golden file，版本控制 |
| 规则冲突 | 误报增加 | 测试覆盖 + code review |
| Go 版本兼容性 | 构建失败 | 锁定 go.mod 版本 |

### 依赖
- golang 1.21+
- 标准 Go testing 包
- 可选：testify, golden 等测试工具
